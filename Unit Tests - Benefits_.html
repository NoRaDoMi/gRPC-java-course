Unit Tests - Benefits?

Published 2 years ago — 6 min read
By huyvha

Before talking about the benefits of unit tests, let's go through the story of our colleagues.

The first story

2 months ago, the company started a new product and Huy is assigned as the main and only one developer. The product has a strict timeline since it has to be released sooner than the competitors.
Because the pressure of "time to market", Huy decided to cut off some processes, including unit testing since he thought his code is good and the unit test is just a waste of time.
He was right, he delivered the product in time and just a few small bugs found by the Tester. The product has been released for 2 months and no additional activities since that time.

Now, Huy is joining another project, still as the main developer. Suddenly, he got a request to improve a feature of the product that he implemented before (the above one).
Unfortunately, he has a "golden fish brain" so when looking at the old code, it seems like another man who implemented it, not him.
Because the product is in production, he starts scaring "do I break something when I modify this function?" and the manager starts wondering "it's just a small modification, why does it take so long to implement, man?".
Okay, he decided to make the changes. He spent hours to test manually the functionalities since he didn't remember the flows and no documentation written in the development phase because of the strict timeline (and this is one of the things he decided to cut off to satisfy the timeline).
He did test the basic flows only because the deadline is coming. Then the changes come to production. He praised day by day to hope no critical bug found and lost 2 kilos because of concerns.
Finally (and luckily), everything went well and he is still one of the best developers in the company (because of the results).

(Life is just a series of lucks...)

The second story

As Huy is currently busy with many new products, Nam is assigned to help Huy on the old ones.
One day, Nam gets the request to implement new features for the old product that Huy did before. Nam starts looking at the code and feels like swimming in the ocean. He did start asking Huy: what is this? what is that? why did you implement like this? does this method work? what is the purpose of this class?...
As Huy is really busy, he answered "please read the code and try to understand the business". Yes, Nam read the code and understand nothing since the code written in rush is the code only understood by God.

Deadline is coming again, Nam did implement the feature but he didn't have any confidence about breaking any working features.
Things come must come, Nam delivers the new features with fears. Things come must come again, but unfortunately, the God of Luck is busy helping Huy, so Nam is alone with lots of bugs found by QA.
QA blames Nam for the poor quality of the product. Nam blames Huy for not supporting him enough, as well as the legacy code is hard to maintain. Huy blames Name for lack of coding skills as well as code reading skills.
Weeks have been spent to fix bugs and test, everyone gets stress...

(The one who comes later is always the poor man...)

So..

After reading the 2 stories, do you find yourself in Huy or Nam?

Do unit tests help with a happier ending?
Maybe yes, maybe no, everything is a tradeoff. But at least, it gives you confidence when you change anything.

Let's imagine in a perfect world, where Huy spent the effort to write unit tests when implementing the product.

  • In the first story, Huy would have confident making any changes since the tests tell him if he breaks something or not and his success would be on his hand instead of the hand of the God of Luck.
  • In the second story, Nam could have been confidently adding/modifying features or refactoring the legacy code when he feels it's hard to maintain.

(When you do things with confident, success will come...)

Theory

Back to the theory, I copy here the benefits of unit testing which I found they are true in my experience:
(source: https://dzone.com/articles/top-8-benefits-of-unit-testing)

  1. Makes the Process Agile
    One of the main benefits of unit testing is that it makes the coding process more Agile. When you add more and more features to a software, you sometimes need to change old design and code. However, changing already-tested code is both risky and costly. If we have unit tests in place, then we can proceed for refactoring confidently.
    Unit testing really goes hand-in-hand with agile programming of all flavors because it builds in tests that allow you to make changes more easily. In other words, unit tests facilitate safe refactoring.
  2. Quality of Code
    Unit testing improves the quality of the code. It identifies every defect that may have come up before code is sent further for integration testing. Writing tests before actual coding makes you think harder about the problem. It exposes the edge cases and makes you write better code.
  3. Finds Software Bugs Early
    Issues are found at an early stage. Since unit testing is carried out by developers who test individual code before integration, issues can be found very early and can be resolved then and there without impacting the other pieces of the code. This includes both bugs in the programmer’s implementation and flaws or missing parts of the specification for the unit.
  4. Facilitates Changes and Simplifies Integration
    Unit testing allows the programmer to refactor code or upgrade system libraries at a later date and make sure the module still works correctly. Unit tests detect changes that may break a design contract. They help with maintaining and changing the code.
    Unit testing reduces defects in the newly developed features or reduces bugs when changing the existing functionality.
    Unit testing verifies the accuracy of the each unit. Afterward, the units are integrated into an application by testing parts of the application via unit testing. Later testing of the application during the integration process is easier due to the verification of the individual units.
  5. Provides Documentation
    Unit testing provides documentation of the system. Developers looking to learn what functionality is provided by a unit and how to use it can look at the unit tests to gain a basic understanding of the unit’s interface (API).
  6. Debugging Process
    Unit testing helps simplify the debugging process. If a test fails, then only the latest changes made in the code need to be debugged.
  7. Design
    Writing the test first forces you to think through your design and what it must accomplish before you write the code. This not only keeps you focused; it makes you create better designs. Testing a piece of code forces you to define what that code is responsible for. If you can do this easily, that means the code’s responsibility is well-defined and therefore that it has high cohesion.
  8. Reduce Costs
    Since the bugs are found early, unit testing helps reduce the cost of bug fixes. Just imagine the cost of a bug found during the later stages of development, like during system testing or during acceptance testing. Of course, bugs detected earlier are easier to fix because bugs detected later are usually the result of many changes, and you don’t really know which one caused the bug.

Last words

Unit testing is a lot like going to the gym. You know it's good for you but finally, you end up with staying home, watching movies, playing games and the reason is "I have no time".

By writing unit tests, you may not see clearly the benefits at the time you write it, but they exist and you or someone else would get them in the future.
